# Phase 1.5: 功能效果确认

## 阶段目标

向用户清晰说明功能的UI效果、操作流程、配置方式和运行时行为，获得明确确认后才能进入开发阶段。

## ⚠️ 核心原则

**未获得用户明确确认，禁止进入Phase 2开发！**

这个阶段的目的是确保：
- ✅ 用户理解功能的最终效果
- ✅ 设计方案符合用户预期
- ✅ 配置方式用户可以接受
- ✅ 没有遗漏的场景

## 必须输出的内容

### 1. UI效果说明

使用ASCII图示或描述清楚界面布局：

```
示例：

┌─────────────────────────┐
│ 面板标题        [关闭]  │
├─────────────────────────┤
│ 字段1: [输入框]         │
│ 字段2: [下拉框]         │
│                         │
│ [保存按钮] [取消按钮]   │
└─────────────────────────┘

说明：
- 面板标题：显示功能名称
- 字段1：配置xxx参数
- 字段2：选择xxx选项
- 保存按钮：保存配置并生效
```

**如果是编辑器工具**：
- 说明在哪个菜单下
- Inspector面板显示哪些字段
- 字段的顺序和分组

**如果是运行时UI**：
- 说明在哪个场景/界面显示
- UI元素的位置和交互方式

### 2. 操作流程说明

清晰描述用户如何使用这个功能：

```markdown
**步骤1：打开配置面板**
- 方式：Unity菜单 -> Tools -> BlockGame -> XXX设置
- 或：在Hierarchy中选择XXXManager，查看Inspector

**步骤2：配置参数**
- 字段1：输入xxx（范围：0-100）
- 字段2：从下拉框选择xxx

**步骤3：保存并应用**
- 点击"保存"按钮
- 配置立即生效（或需要重启场景）

**步骤4：验证效果**
- 运行游戏
- 观察xxx行为是否符合预期
```

### 3. 配置示例（至少2个）

提供2-3个典型使用场景的配置示例：

```markdown
**示例1：新手引导场景**
- 字段1：10（低难度）
- 字段2：Tutorial（引导模式）
- 目的：为新手提供简单的游戏体验
- 效果：游戏难度降低，显示提示信息

**示例2：挑战模式**
- 字段1：80（高难度）
- 字段2：Challenge（挑战模式）
- 目的：为高级玩家提供挑战
- 效果：游戏难度提升，限时完成

**示例3：测试模式**
- 字段1：0（禁用）
- 字段2：Debug（调试模式）
- 目的：开发者测试功能
- 效果：显示调试信息，跳过限制
```

### 4. 运行时行为说明

说明不同配置下的运行时表现：

```markdown
**场景1：配置了字段1 = 50**
- 初始化：系统启动时加载配置
- 运行时：每次xxx时应用此配置
- 结果：xxx行为按50的强度执行

**场景2：未配置字段1（默认值）**
- 初始化：使用默认值 = 30
- 运行时：按默认配置执行
- 结果：标准行为

**场景3：字段1超出范围**
- 验证：保存时检查范围
- 处理：显示警告，使用最近的有效值
- 结果：字段1被限制在0-100范围内

**场景4：场景切换时**
- 行为：配置保持（或重置）
- 原因：XXXManager是跨场景单例（或场景级别单例）
- 注意：如果需要重置，在场景加载时调用Reset()
```

## 用户响应处理

### 如果用户确认无误
✅ 更新TodoWrite，标记Phase 1.5完成
✅ 进入Phase 2开发阶段

### 如果用户提出修改
⚠️ 返回Phase 1架构设计，调整设计方案
⚠️ 如果涉及重大变更，重新执行Phase 1.25设计质询
⚠️ 修改后重新进行Phase 1.5确认

### 如果用户不确定
❓ 针对不确定的部分，提供更多选项
❓ 说明不同选择的优缺点
❓ 给出推荐方案并说明理由

## 常见错误

❌ **过于简略，用户看不懂效果**
- 后果：用户无法判断是否符合需求，开发后需要大量返工

❌ **跳过配置示例**
- 后果：用户不知道如何使用，或配置错误

❌ **不说明边界情况和异常处理**
- 后果：运行时出现意外行为，用户认为是Bug

❌ **没有等用户确认就开始开发**
- 后果：方向错误，浪费大量时间
