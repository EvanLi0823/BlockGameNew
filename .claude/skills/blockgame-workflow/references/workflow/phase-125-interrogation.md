# Phase 1.25: 设计质询

## 阶段目标

在设计完成后、开发实现前，对设计方案进行系统化质询，识别潜在风险、Bug隐患、边界情况，确保设计方案的完整性和正确性。

## 何时使用

### ✅ 必须使用（强制执行）

| 任务类型 | 原因 | 质询重点 |
|---------|------|----------|
| **架构重构项目** | 影响面大，风险高 | 向后兼容性、迁移风险、回滚方案 |
| **新系统设计** | 从零开始，容易遗漏 | 模块交互、数据流设计、扩展性 |
| **复杂功能开发** | 涉及多个模块 | 数据迁移、多系统交互、状态管理 |

### 🟡 推荐使用

| 任务类型 | 原因 | 质询重点 |
|---------|------|----------|
| **中等复杂度功能** | 涉及2-3个模块交互 | 接口设计、依赖关系、异常处理 |
| **复杂Bug修复** | 根因复杂或影响面大 | 根因分析、副作用评估、回归测试 |
| **性能优化方案** | 可能引入副作用 | 性能收益、稳定性影响、可监控性 |

### ⚪ 可跳过

| 任务类型 | 原因 | 注意事项 |
|---------|------|----------|
| **简单UI调整** | 纯视觉效果修改 | 确保不影响逻辑 |
| **参数配置修改** | 单一配置文件调整 | 确保配置生效 |
| **紧急热修复** | 时间紧迫 | **事后必须补充质询** |

## 💡 设计质询的价值

| 价值点 | 说明 | 数据支撑 |
|--------|------|----------|
| ✅ **设计阶段发现问题** | 成本最低，改动最小 | 提前发现率 +60% |
| ✅ **减少实现阶段返工** | 避免大量代码重写 | 返工率 -50% |
| ✅ **提升用户确认质量** | 需求对齐更精准 | 需求变更 -40% |
| ✅ **识别边界情况** | 覆盖异常场景 | 边界场景覆盖 +40% |
| ✅ **评估系统影响** | 向后兼容性保障 | 回归Bug -30% |

## 使用流程

### 步骤1：调用 design-interrogator 技能

```bash
# 触发设计质询技能
/skill design-interrogation

# 或在对话中明确说明
"请使用 design-interrogator 代理审查这个设计方案"
```

### 步骤2：提供设计文档

将Phase 1的架构设计文档提供给质询代理：
- 数据结构设计
- 接口和调用流程
- 架构图/流程图
- 配置方案

### 步骤3：接受多维度质询

design-interrogator 会从以下维度进行质询：

**维度1: 实现方案审查**
- 如何实现？使用哪些组件/技术？
- 为什么这样实现？有其他方案吗？
- 实现细节：数据结构、生命周期、API设计

**维度2: 系统影响分析**
- 向后兼容性：现有功能是否受影响？
- 性能影响：是否有性能瓶颈？
- 依赖变化：模块依赖关系变化？
- 数据迁移：旧数据如何处理？

**维度3: Bug风险识别**
- 空引用风险：NullReferenceException可能性
- 时序问题：初始化顺序、生命周期
- 并发问题：协程/异步竞争条件
- 边界条件：空值、极值、异常场景

**维度4: 边界情况验证**
- 数据边界：null、空集合、极端数值
- 时序边界：未初始化、场景切换、暂停/恢复
- 状态边界：已销毁对象、禁用GameObject
- 平台边界：分辨率、性能、版本

**维度5: 可维护性审查**
- 代码可读性：命名、注释
- 可测试性：验证方法、调试便利性
- 可扩展性：未来需求扩展
- 配置灵活性：可配置参数

### 步骤4：获取质询报告

design-interrogator 会输出质询报告，包含：

```markdown
# 设计质询报告

## 🔴 高优先级问题（必须解决）
1. [问题描述]
   - 风险：...
   - 建议：...

## 🟡 中优先级问题（建议解决）
1. [问题描述]
   - 影响：...
   - 建议：...

## 🟢 低优先级建议（可选优化）
1. [建议描述]
   - 收益：...
   - 成本：...

## 📋 关键确认问题清单
- [ ] 问题1：...
- [ ] 问题2：...

## 🎯 后续行动建议
1. 修改设计方案，解决高优先级问题
2. 在Phase 1.5中向用户确认关键问题
3. 在Phase 2开发时注意边界情况处理
```

### 步骤5：处理质询结果

**对于高优先级问题**：
- ⚠️ 必须返回Phase 1修改设计方案
- ⚠️ 修改后重新进行质询

**对于中优先级问题**：
- 💡 评估成本和收益
- 💡 决定是否修改设计或在开发时处理

**对于低优先级建议**：
- 📝 记录在案
- 📝 可在后续优化阶段处理

### 步骤6：整合到Phase 1.5

将质询报告的**关键确认问题清单**整合到Phase 1.5的用户确认中：
- 向用户说明质询发现的问题
- 询问用户的选择和偏好
- 确保用户理解风险和限制

## 输出物

- ✅ 质询报告（高/中/低优先级问题）
- ✅ 关键确认问题清单
- ✅ 后续行动建议
- ✅ 边界场景测试用例

## 常见错误

❌ **跳过质询，直接开发**
- 后果：开发时发现设计缺陷，大量返工

❌ **忽略高优先级问题**
- 后果：上线后出现严重Bug

❌ **质询报告不整合到Phase 1.5**
- 后果：用户不知道风险，确认流于形式

❌ **紧急热修复后不补充质询**
- 后果：技术债累积，后续维护困难

## 质询与审查的区别

| 阶段 | Phase 1.25 设计质询 | Phase 3 代码审查 |
|------|-------------------|-----------------|
| **时机** | 设计完成后，开发前 | 开发完成后，提交前 |
| **对象** | 设计文档、架构方案 | 实现代码 |
| **目的** | 发现设计缺陷和遗漏 | 检查代码质量和规范 |
| **成本** | 修改成本低（未开发） | 修改成本高（已开发） |
| **重点** | 方案完整性、风险识别 | 代码质量、规范合规 |
| **工具** | design-interrogator | unity-reviewer |

## 实践案例

### 案例1：新增货币系统（复杂功能）

**Phase 1设计**：
- 新增CurrencyManager
- 支持多种货币类型
- 货币转换和兑换

**Phase 1.25质询发现**：
- 🔴 高优先级：现有存储系统不支持多货币，需要数据迁移方案
- 🟡 中优先级：货币转换汇率如何配置？是否支持动态调整？
- 🟢 低优先级：建议添加货币历史记录功能

**处理**：
- 返回Phase 1，设计数据迁移方案
- 在Phase 1.5向用户确认汇率配置方式
- 记录历史记录功能，留待后续版本

**收益**：
- 避免了数据迁移的坑（节省2天返工）
- 提前确认汇率配置，避免需求变更

### 案例2：UI调整（简单任务，跳过质询）

**Phase 1设计**：
- 修改按钮颜色和大小
- 调整布局间距

**是否质询**：⚪ 跳过
- 原因：纯视觉调整，不涉及逻辑
- 直接进入Phase 1.5确认效果
